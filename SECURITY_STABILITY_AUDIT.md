# Security & Stability Audit

Дата аудита: 2026-04-27

## Критические риски (P0/P1)

### 1) Некорректная модель доверия к IP (spoofing через `X-Forwarded-For`)
- **Где**: `app.py` берет IP из `X-Forwarded-For` без валидации доверенного прокси.
- **Риск**: злоумышленник может подменять IP, обходить rate limit / throttling и искажать аналитику.
- **Фикс**:
  1. Доверять `X-Forwarded-For` только от известных reverse proxy (Nginx/Cloudflare).
  2. Иначе использовать `self.client_address[0]`.
  3. Добавить парсер цепочки `X-Forwarded-For` + allowlist прокси-сетей.

### 2) reCAPTCHA проверяется не полностью
- **Где**: проверяется `success` и `score`, но не проверяются `action` и `hostname`.
- **Риск**: возможен replay/подмена токена с другого action/домена.
- **Фикс**:
  1. Проверять `action == "lead_submit"`.
  2. Проверять `hostname` против прод-домена.
  3. Ввести fail-closed политику для production.

### 3) PII в логах без маскирования/ретеншна
- **Где**: в `data/leads.jsonl` пишутся телефон, IP и User-Agent.
- **Риск**: утечка персональных данных, регуляторные и репутационные риски.
- **Фикс**:
  1. Маскировать телефон (например, `+7999***0000`) в оперативных логах.
  2. Отдельно хранить сырой лид в защищенном хранилище.
  3. Ввести ротацию/TTL логов, шифрование тома и ограничение прав доступа.

### 4) Webhook retry без гарантированной идемпотентности на стороне CRM
- **Где**: есть `X-Lead-ID`, но не описано обязательное idempotency-поведение приемника.
- **Риск**: дубли лидов при частичном успехе (таймаут после приема).
- **Фикс**:
  1. Зафиксировать контракт: CRM обязана дедуплицировать по `X-Lead-ID`/`lead_id`.
  2. Добавить локальный дедуп-кэш на короткое окно.
  3. Логировать `delivery_status` и `attempt_no`.

## Высокие риски (P1/P2)

### 5) Ограничения rate limit — in-memory, не shared
- **Где**: `defaultdict(deque)` в процессе Python.
- **Риск**: при горизонтальном масштабировании лимиты не согласованы; после рестарта «обнуляются».
- **Фикс**:
  1. Перенести лимиты в Redis (token bucket / sliding window).
  2. Вести отдельные лимиты по IP + fingerprint (IP/UA).

### 6) Синхронные внешние вызовы на критическом пути
- **Где**: webhook/уведомления выполняются синхронно в запросе.
- **Риск**: рост latency, таймауты клиента, снижение пропускной способности.
- **Фикс**:
  1. Отдавать `202 Accepted` после сохранения лида.
  2. Доставку в webhook/уведомления вынести в фоновую очередь (RQ/Celery/worker).

### 7) Слабая валидация входных данных
- **Где**: обязательность проверяется только для `phone` как non-empty.
- **Риск**: мусорные лиды, инъекции в downstream-системы.
- **Фикс**:
  1. Ввести схему (Pydantic/JSON schema): длины, типы, allowlist символов.
  2. Нормализовать телефон в E.164.
  3. Отсекать поля сверх лимита длины.

### 8) Отсутствие секрет-менеджмента
- **Где**: секреты задаются env-переменными без указания безопасного источника.
- **Риск**: утечка секретов через shell history/CI logs.
- **Фикс**:
  1. Использовать Secret Manager (Docker secrets, Vault, cloud secret stores).
  2. Исключить вывод секретов в логи и диагностические дампы.

## Средние риски (P2)

### 9) Недостаток observability
- **Где**: нет метрик/алертов уровня приложения.
- **Риск**: позднее обнаружение деградации и потерь лидов.
- **Фикс**:
  1. Метрики: success rate, webhook retry count, notify fail rate, p95 latency.
  2. Алерты: spike 4xx/5xx, zero-lead window, webhook failures.

### 10) OG-баннер и контент готовности
- **Где**: проверка размеров есть, в репозитории используется текстовый `og-cover.svg`.
- **Риск**: снижение CTR и отклонения в рекламных системах.
- **Фикс**:
  1. Поставить финальный `og-cover.svg` (или генерируемый из него `og-cover.jpg`) 1200x630.
  2. Прогнать Telegram/Facebook Debugger и зафиксировать скриншоты в release checklist.

## Рекомендованный план исправлений

1. **P0/P1 (сразу):** доверенный IP, полноценная reCAPTCHA-проверка (`action/hostname`), PII-стратегия, контракт идемпотентности CRM.
2. **P1/P2:** Redis rate limit + очередь доставки webhook/уведомлений.
3. **P2:** observability, security hardening (secrets, retention policy), контент/OG final QA.

## Минимальный Definition of Done перед трафиком

- Все smoke-тесты проходят.
- Проверка OG (`1200x630`) проходит.
- Включен контракт дедупликации в CRM по `lead_id`.
- Подтверждены live-события в GA4/GTM/Meta/Ads.
- Настроены алерты на потери лидов и ошибки доставки.
